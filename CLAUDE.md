## Go Code Style

- All code must be gofumpt-clean.
- Lines up to 180 chars are fine — don't split lines that fit within 180 chars.
- Beyond 180 chars, fully expand — one argument / field / element per line, trailing comma
  on every line including the last, closing paren/bracket/brace on its own line. No
  half-splits (where some args share a line). This applies to function calls, function
  signatures, struct literals, and slice/map literals.
- Map literals with more than one entry should always be split (one entry per line),
  regardless of line length.
- Three import groups: stdlib, third-party, internal calico packages.
- Short receiver names: single letter matching the type.
- `fmt.Errorf` with `%w` (not `%v`) for error wrapping.
- `logrus.WithError(err)` for error logging, not inlined in format strings.
- Comments explain "why", not "what". No comments on self-documenting code.
- Doc comments start with the exported name: `// Foo does...`
- Always use safe type assertions (`obj, ok := x.(*T)`) — never bare assertions that can panic.
- Never ignore error returns. Always check and handle them, even from cache lookups or seemingly infallible calls.

## Git / PR Workflow

- When asked to work on a PR or branch, and the local repo either has pending changes you didn't make or is not on master, perform the work in a new temporary git worktree under `/tmp` (e.g., `git worktree add /tmp/wt-<description> <branch>`). After pushing and creating the PR, clean up with `git worktree remove /tmp/wt-<description>` (use `--force` if needed for untracked files).
- Push branches to my personal fork (`cd4` remote) for: `projectcalico/*`,
  `tigera/operator`. For other `tigera/*` repos, push to origin. For any other
  repos, check local git remotes and prefer my personal fork if it exists and
  is pushable.
- Branch names: use the format `casey-<2-3 word description>` (e.g., `casey-flowtable-support`, `casey-webhook-manifest`, `casey-ifupdate-race`).
- Create new commits rather than amending unless I explicitly ask.
- Never force push without asking first.
- Never skip hooks (--no-verify).
- PR descriptions: no `## Summary` or `## Test plan` headers — jump straight into the
  content. Use `## Description` for longer PRs. No AI attribution.
- Commit messages: short imperative summary, brief "why" explanation if needed.
- PR labels (calico, calico-private, operator): PRs without user-facing impact (tests, refactors, etc) get `docs-not-required` and `release-note-not-required` labels. PRs with user-facing impact get appropriate docs label and a release note in a ` ```release-note ` block in the PR body.

## Cross-Repo Merge Considerations

- When planning changes to `projectcalico/calico`, consider the implications for merging to `tigera/calico-private` as part of the design (e.g., avoid unnecessary conflicts, keep changes modular).
- When making changes in `tigera/calico-private`, consider the reverse — how will upstream merges from `projectcalico/calico` interact with these changes.

## Calico-Specific

- CRD generation: `make -C api gen-files` regenerates CRDs, deep copies, client sets.
- CEL XValidation rules: always set MaxLength/MaxItems on fields used in CEL expressions.
  Use `size(self.field) == 0` instead of `self.field == ''` (goimports corrupts single quotes).
- E2e tests: see the e2e-tests skill for conventions. Key points:
  - Tests run in parallel by default (`ginkgo -procs=4`). Write parallel-safe tests.
  - `WithSerial()` ONLY for truly global state (FelixConfiguration, BGPConfiguration, node labels).
  - Scope GlobalNetworkPolicies to test namespace via `NamespaceSelector`, don't use `Selector: "all()"`.
  - `utils.GenerateRandomName()` for all cluster-scoped resource names.
  - Do NOT use `CleanDatastore` unless the test is Serial.
  - `DeferCleanup` per resource, not grouped deletes.
  - `Eventually` returns `error`, not `bool`.
  - Prefer existing product client libraries over hand-rolling HTTP.
  - Enterprise-only code uses `_enterprise.go` suffix (tests and utilities).
